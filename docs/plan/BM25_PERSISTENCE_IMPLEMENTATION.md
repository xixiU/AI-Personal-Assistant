已完成 BM25 索引持久化功能，解决应用启动时每次重建 BM25 索引的性能瓶颈。

#### 2.2 文档指纹计算（第 482-492 行）
```python

    """计算文档集合的指纹（用于检测文档变化）"""
```
- 使用文档 token 列表的 MD5 hash 作为指纹
- 用于检测文档集合是否变化

#### 2.3 索引持久化（第 494-522 行）
```python
def _save_bm25_index(self, docs: List[Dict[str, str]]) -> None:
    """持久化 BM25 索引到磁盘"""
```
- 使用 `pickle` 序列化 BM25Okapi 对象
- 保存内容：
  - `bm25`: BM25Okapi 对象
  - `bm25_docs`: 原始文档列表
  - `bm25_corpus`: 分词后的语料库
  - `docs_hash`: 文档集合指纹
- 异常处理：持久化失败不影响正常使用（降级）

#### 2.4 索引加载（第 524-566 行）
```python
def _load_bm25_index(self, docs: List[Dict[str, str]]) -> bool:
    """从磁盘加载 BM25 索引"""
```
- 检查索引文件是否存在
- 加载并校验文档指纹（hash 匹配）
- 失效或失败时返回 False，触发重建
- 异常处理：加载失败静默回退到重建

#### 2.5 索引构建流程优化（第 317-436 行）
修改 `index_documents` 方法：
```python
# 1. 尝试加载已有索引
bm25_loaded = self._load_bm25_index(docs)

# 2. 向量索引（ChromaDB 自己管理持久化）
...

# 3. BM25 索引：仅在未加载成功时重建
if not bm25_loaded:
    # 构建 BM25 索引
    ...
    # 持久化
    self._save_bm25_index(docs)
else:
    bm25_time = 0  # 从缓存加载，不计入构建时间
```

### 3. 关键设计决策

#### 3.1 为什么使用 pickle？
- 标准库，无额外依赖
- 序列化速度快
- BM25Okapi 对象支持 pickle

#### 3.2 为什么用 MD5 hash 而不是时间戳？
- 时间戳无法检测文档内容变化（只修改不增删）
- Hash 基于 token 列表，精确检测文档集合变化
- Token 排序后计算，避免顺序差异

#### 3.3 为什么失败时静默回退？
- BM25 索引可以实时重建
- 持久化是性能优化，不是核心功能
- 避免因持久化失败导致应用无法启动

#### 3.4 为什么不持久化向量索引？
- ChromaDB 已自行管理持久化（`PersistentClient`）
- 任务范围明确：只优化 BM25 部分

### 4. 性能预期

#### 4.1 冷启动（首次构建）
- 构建 BM25 索引：数秒到数十秒（取决于文档数量）
- 持久化耗时：< 1 秒（通常 < 0.5 秒）
- 索引文件大小：< 文档总大小的 20%

#### 4.2 热启动（从缓存加载）
- 加载耗时：< 1 秒（通常 < 0.2 秒）
- 性能提升：**10x-50x** 倍（相比重建）
- 对于 100 篇文档：重建 5 秒 → 加载 0.2 秒

#### 4.3 增量更新
- 检测到文档变化时自动重建
- 重建后立即持久化
- 下次启动仍可快速加载

### 10. 潜在风险与缓解

#### 10.1 Pickle 安全性
- **风险**：pickle 反序列化可能执行恶意代码
- **缓解**：索引文件仅由应用自己生成，不接受外部输入

#### 10.2 索引文件损坏
- **风险**：磁盘故障或进程崩溃导致文件损坏
- **缓解**：加载失败时自动回退到重建，不影响应用运行

#### 10.3 Hash 冲突
- **风险**：MD5 hash 理论上可能冲突
- **缓解**：基于 token 列表（短字符串），实际冲突概率极低

#### 10.4 磁盘空间
- **风险**：索引文件占用额外磁盘空间
- **缓解**：索引文件大小合理（< 文档总大小的 20%）
