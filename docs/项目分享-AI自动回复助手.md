## 一、一句话项目介绍

> 💡 **在飞书群里 @ 机器人提问，它能基于团队私域知识库（飞书知识库 / 云空间 / 本地文档）智能回答。**

```
┌─────────┐   @ 机器人    ┌─────────┐   检索文档    ┌──────────┐
│  用户   │ ───────────▶ │ AI 助手  │ ───────────▶ │ 知识库   │
│ 飞书群  │ ◀─────────── │ (本系统) │ ◀─────────── │ 飞书+本地│
└─────────┘   智能回答    └─────────┘   返回内容    └──────────┘
                              │
                              ▼
                       ┌──────────────┐
                       │ 大模型       │
                       │ Claude/GPT/  │
                       │ Dify/Ollama  │
                       └──────────────┘
```

## 二、解决什么问题

| 痛点 | 现状 | 本项目方案 |
|------|------|-----------|
| 🔍 文档查找慢 | 飞书知识库手动搜索，关键词不准就找不到 | 自然语言提问，混合检索（语义+关键词）一步定位 |
| 👨‍🏫 新人答疑重复 | 群里反复问同样问题，老同事疲于解答 | 机器人 7×24 答复，链路文档作为依据 |
| 📚 知识沉淀难用 | 文档写了没人看，还在群里追着问 | 文档自动同步到机器人，问答即用 |
| 🔄 多种工具切换 | 飞书查文档 → 切换 ChatGPT 总结 → 复制回飞书 | 一站式，机器人内嵌 LLM + 知识检索 |

## 三、四大核心能力

### 🤖 能力一：飞书机器人原生接入

> ✅ **官方 API + 事件驱动**，不依赖客户端 hack，稳定可控

- 支持 **私聊 / 群聊 / @ 提及** 三种触发场景
- **群组白名单 / 用户白名单**，灵活管控影响范围
- 自动 **入群欢迎消息**，新成员第一时间知道机器人能干啥
- 飞书要求 **3 秒内响应 webhook**，已实现异步队列保证 < 200ms 响应

### 📚 能力二：私域知识库 RAG（项目核心亮点）

> ✅ **混合检索 + 中文 Embedding**，让 AI 真正"看得懂"团队文档

支持三种数据源：
| 来源 | 接入方式 | 适用场景 |
|------|---------|---------|
| 📖 飞书知识库 | MCP 协议 + 增量同步 | 团队知识库（产品文档、流程规范） |
| ☁️ 飞书云空间 | MCP 协议 + 文件夹递归 | 项目文档、临时材料 |
| 💾 本地文档 | 目录扫描（md/txt/sql/yaml） | SQL 配置、产品手册等不便上传的内容 |

**检索策略**：

```
用户提问
   │
   ├─▶ 向量检索（ChromaDB + text2vec-base-chinese）  ─┐
   │      └ 语义匹配，找"意思相近"的文档              │
   │                                                  ├─▶ RRF 融合排序 ──▶ 标题相关性过滤 ──▶ 注入 Prompt
   └─▶ BM25 检索（jieba 分词）                       ─┘
          └ 关键词精确匹配，找"字面相同"的文档
```

**关键设计**：
- 🔄 **增量更新**：基于飞书文档 `edit_time` 比对，只拉变更的，避免全量同步
- 📂 **保留目录结构**：本地缓存按飞书原始路径组织，便于排查
- 🔗 **附带原文链接**：回答时给出飞书原文 URL，可一键跳转
- ✂️ **大文档分块**：> 2000 字符自动分块索引，提升 Embedding 质量

### 🔌 能力三：多 AI 模型一键切换

> ✅ **一份配置，自由切换 4 大主流 LLM**

| Provider | 适用场景 | 配置示例 |
|----------|---------|---------|
| **Anthropic Claude** | 长上下文、文档问答效果最优 | `provider: anthropic` |
| **OpenAI 兼容** | GPT / Ollama / LM Studio / Azure | `provider: openai` |
| **Dify** | 已有 Dify 应用，直接对接 | `provider: dify` |
| **本地模型** | 数据敏感，私有化部署 | Ollama + `provider: openai` |

**统一调用入口**（基类封装）：所有 Provider 自动获得调用日志、耗时统计、token 计数，无需重复埋点。

### 🚀 能力四：生产级部署能力

> ✅ **Docker 一键部署 + GPU 加速 + 高并发**

| 维度 | 实现 |
|------|-----|
| 🐳 容器化 | Docker + docker-compose，CPU/GPU 双版本 |
| ⚡ GPU 加速 | ONNX Runtime CUDA，**Embedding 速度 5-10×**；支持多卡轮询 |
| 🔀 高并发 | 事件队列 + 线程池，**支持多用户同时提问** |
| 🩺 可观测 | Trace ID 全链路 + 心跳日志 + 性能埋点（CPU/内存/耗时）|
| 🛡️ 健壮性 | 信号处理、全局异常捕获、自动重试、graceful shutdown |
| 📦 一键运维 | `service.sh start/stop/restart/status/debug/monitor` |

## 四、技术架构

### 整体架构图

```
┌──────────────────────────────────────────────────────────────────┐
│                       飞书开放平台（Webhook 推送）                │
└────────────────────────────┬─────────────────────────────────────┘
                             │ HTTPS
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│  Flask + Waitress (生产级 WSGI)                                  │
│  ├─ /webhook/feishu   → 事件接收（< 200ms 响应）                 │
│  ├─ /api/chat         → Web 测试接口                             │
│  └─ /health           → 健康检查                                 │
└────────────────────────────┬─────────────────────────────────────┘
                             │ enqueue
                             ▼
                   ┌────────────────────┐
                   │   事件队列 (Queue) │   ← 削峰、异步解耦
                   └─────────┬──────────┘
                             │ dispatch
                             ▼
                   ┌────────────────────┐
                   │  线程池 (5 worker) │   ← 多用户并发
                   └─────────┬──────────┘
                             │
                ┌────────────┼────────────┐
                ▼            ▼            ▼
        ┌───────────┐ ┌───────────┐ ┌───────────┐
        │ Context   │ │  AI       │ │ 文档检索  │
        │ Manager   │ │ Provider  │ │ 引擎      │
        │(会话历史) │ │(LLM 调用) │ │(混合检索) │
        └───────────┘ └───────────┘ └─────┬─────┘
                                          │
                          ┌───────────────┼───────────────┐
                          ▼               ▼               ▼
                   ┌──────────┐   ┌──────────┐   ┌──────────┐
                   │ChromaDB  │   │  BM25    │   │ MCP 客户 │
                   │(向量库)  │   │(关键词)  │   │端(飞书)  │
                   └──────────┘   └──────────┘   └──────────┘
```

### 核心技术选型

| 层级 | 选型 | 选型理由 |
|------|------|---------|
| Web 框架 | Flask + Waitress | Flask 轻量，Waitress 是 Windows/Linux 通用的生产级 WSGI |
| Embedding | text2vec-base-chinese (ONNX) | 中文场景强于多语言模型；ONNX 比 PyTorch 体积小 10×、启动快 |
| 向量库 | ChromaDB | 嵌入式、零运维、社区活跃，足以支撑万级文档 |
| 关键词检索 | rank-bm25 + jieba | 中文场景关键词比向量更可靠（专有名词、ID 等） |
| 融合算法 | RRF (Reciprocal Rank Fusion) | 经典融合算法，无需调权重，BM25 加权 1.5 偏向关键词 |
| 飞书集成 | MCP 协议 (SSE) | 官方推荐协议，支持知识库 + 云空间双源 |
| 容器 | Docker + nvidia/cuda 镜像 | 解决旧 glibc 兼容问题；CPU/GPU 镜像分离 |
| 日志 | loguru + trace_id (contextvars) | 多线程下用 contextvars 维持 trace，日志全链路可追 |

## 六、性能数据

### 📊 检索性能

| 场景 | CPU 模式 | GPU 模式（单卡 P40）| 加速比 |
|------|---------|---------------------|--------|
| 索引 1000 篇文档 | ~120s | ~15s | **8×** |
| 单次查询 | ~500ms | ~100ms | **5×** |
| 批量 Embedding (batch=128) | OOM 风险 | < 2GB 显存 | - |

### 📊 端到端响应

| 阶段 | 耗时 | 说明 |
|------|-----|------|
| Webhook 接收 → 入队 | < 50ms | Flask + Waitress |
| 文档检索（命中缓存） | 100-300ms | 向量 + BM25 并行 |
| LLM 调用（Claude Sonnet） | 2-5s | 取决于上下文长度 |
| 飞书回复 API | < 500ms | 官方 API |
| **总计** | **3-6s** | 用户感知良好 |

### 📊 并发能力

- 默认 **5 个 worker 线程**，可配置
- 队列 100 容量，自动削峰
- 实测同时 10 用户提问，平均响应 4.2s（无队列堆积）

## 七、典型应用场景

### 💼 场景一：团队 FAQ 自动答疑
- **痛点**：群里反复问"X 系统的部署文档在哪""Y 接口怎么调"
- **解法**：把文档丢进飞书知识库，机器人自动答 + 给原文链接
- **收益**：老同事少打断，新人即问即得

### 💼 场景二：新人 onboarding
- **痛点**：新人入职 1 周问的问题 80% 都重复
- **解法**：把入职手册、规范文档加入知识库，新人 @ 机器人自助查询
- **收益**：HR / Mentor 减负，新人响应不卡顿

### 💼 场景三：技术支持智能客服
- **痛点**：客户问的产品问题、配置问题分散在群里没沉淀
- **解法**：产品文档 + SQL 配置作为本地文档接入，客户群部署机器人
- **收益**：标准答案统一、可追溯

### 💼 场景四：运维 SOP 查询
- **痛点**：故障处理 SOP 散落在多处，紧急时找不到
- **解法**：SOP 文档纳入知识库，运维群提问"X 服务挂了怎么办"
- **收益**：MTTR 缩短

## 八、安全与合规

| 维度 | 设计 |
|------|------|
| 🔐 配置安全 | API Key 不入 Git；`config.yaml` 强制 .gitignore |
| 🔒 数据隐私 | 消息**仅本地处理**；日志不记录完整内容（仅长度+摘要）|
| 🛡️ 非侵入式 | 仅用官方 API；**不修改 IM 客户端、不 hook 系统 API** |
| ✅ 权限可控 | 群组/用户白名单；用户可随时停用 |
| 🌐 网络隔离 | 支持禁用系统代理；支持私有化大模型（Ollama 等） |
