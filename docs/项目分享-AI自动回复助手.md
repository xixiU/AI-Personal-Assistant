## 一、项目是什么

> 一个支持多渠道接入的 AI 智能问答系统，能基于团队私域知识库（飞书知识库 / 云空间 / 本地文档）智能回答，并附带原文链接。

支持三种使用方式：
- **飞书机器人**：群里 @ 机器人或私聊提问（Webhook 事件驱动）
- **微信**：监听微信窗口消息，自动回复（GUI 轮询方式）
- **Web 页面**：直接浏览器访问，不依赖任何 IM 工具

```
                    ┌─────────┐
                    │ 飞书群  │──── Webhook API ────┐
                    └─────────┘                     │
                    ┌─────────┐                     ▼
                    │  微信   │──── GUI 轮询 ──▶ ┌─────────┐   混合检索   ┌──────────────┐
                    └─────────┘                  │ AI 助手  │ ──────────▶ │ 知识库       │
                    ┌─────────┐                  │ (本系统) │ ◀────────── │ 飞书+本地文档│
                    │ Web 页面│──── HTTP API ──▶ └─────────┘              └──────────────┘
                    └─────────┘                      │
                                                    ▼
                                             ┌──────────────┐
                                             │ 大模型       │
                                             │ Claude/GPT/  │
                                             │ Dify/Ollama  │
                                             └──────────────┘
```

## 二、痛点与解决方案

### 核心痛点

我们团队日常遇到的问题：

**1. 文档找不到、找不准**

现状：飞书知识库有几十上百篇文档，但搜索只支持关键词精确匹配。用户不知道文档标题叫什么，搜不到就放弃了。

解决：混合检索引擎（语义向量 + BM25 关键词），用自然语言提问就能定位到相关文档段落。

**2. 重复答疑消耗人力**

现状：群里反复问"X 系统怎么部署""Y 接口参数是什么"，老同事每次都要手动翻文档回答。

解决：机器人 7×24 自动答复，回答附带原文链接，用户可以自行深入阅读。

**3. 知识沉淀了但没人用**

现状：文档写了放在知识库里，但大家习惯在群里直接问人。文档越写越多，利用率越来越低。

解决：把知识库接入机器人，问答即用。文档更新后自动同步，不需要额外维护。

**4. 工具链割裂**

现状：飞书查文档 → 切到 ChatGPT 总结 → 复制回飞书回复。三步操作，上下文丢失。

解决：一站式。机器人内嵌大模型 + 知识检索，在飞书内完成全部流程。

## 三、技术选型与实现路径

### 为什么不直接用 Dify / Coze / FastGPT？

| 对比维度 | SaaS 平台 | 本项目 |
|----------|----------|--------|
| 飞书私有化部署 | 不支持或需要额外适配 | 原生支持（讯飞飞书等私有化环境） |
| 数据安全 | 文档上传到第三方 | 数据不出内网，本地处理 |
| 模型自由度 | 绑定平台模型 | 一份配置切换 Claude/GPT/Ollama/Dify |
| 定制能力 | 受限于平台功能 | 完全可控，按需扩展 |
| 成本 | 按量付费，规模上去后贵 | 自部署，模型费用可控 |

### 核心技术选型

| 层级 | 选型 | 为什么选它 |
|------|------|-----------|
| Web 框架 | Flask + Waitress | Flask 轻量上手快；Waitress 是跨平台生产级 WSGI，不依赖 Linux |
| Embedding | text2vec-base-chinese (ONNX) | 中文语义理解强于多语言模型；ONNX 格式比 PyTorch 体积小 10×、启动快、支持 GPU |
| 向量库 | ChromaDB | 嵌入式零运维，万级文档够用，不需要额外部署 ES/Milvus |
| 关键词检索 | rank-bm25 + jieba | 专有名词、ID、版本号等场景，关键词比语义更可靠 |
| 融合算法 | RRF (Reciprocal Rank Fusion) | 经典融合算法，无需调权重，BM25 加权 1.5 偏向精确匹配 |
| 飞书文档接入 | MCP 协议 (SSE) | Anthropic 推荐的工具协议，支持知识库 + 云空间双源 |
| 容器化 | Docker + NVIDIA CUDA 镜像 | 解决服务器 glibc 版本低的兼容问题；CPU/GPU 镜像分离 |
| 日志追踪 | loguru + contextvars | 多线程下用 contextvars 维持 trace_id，全链路日志可追溯 |

### 实现路径（从 0 到生产）

```
第一步：跑通链路
├── 飞书 Webhook 接入（事件订阅 + 加密验证）
├── 消息解析 → AI 调用 → 飞书回复
└── 验证：群里 @ 机器人能收到回复

第二步：并发架构
├── 事件队列 + 线程池（解决多用户排队阻塞）
├── Trace ID 全链路（contextvars，类似 Java MDC）
└── 验证：10 人同时提问不卡顿

第三步：知识库 RAG
├── 飞书 MCP 文档拉取 + 本地缓存
├── 混合检索引擎（向量 + BM25 + RRF 融合）
├── 中文 Embedding (text2vec-base-chinese ONNX)
└── 验证：问文档里的内容能准确回答

第四步：生产化
├── Docker 部署（CPU + GPU 双镜像）
├── GPU 加速 Embedding（ONNX Runtime CUDA）
├── 服务管理脚本（start/stop/monitor/自动重启）
└── 验证：服务稳定运行 7×24
```

## 四、当前具备的能力

### 多渠道接入

| 渠道 | 接入方式 | 特点 |
|------|---------|------|
| 飞书机器人 | 官方 Webhook API + 事件驱动 | 私聊/群聊/@提及，白名单管控，入群欢迎消息 |
| 微信 | GUI 轮询（pywinauto/pywechat） | 监听指定聊天窗口，自动提取消息并回复 |
| Web 页面 | 内置前端 + `/api/chat` 接口 | 浏览器直接访问，不依赖任何 IM 工具，支持图片上传 |

- 飞书要求 3 秒内响应 webhook，已实现异步队列 < 200ms 响应
- 微信支持 3.9 和 4.1 两个版本，通过配置切换
- Web 页面内置在服务中，启动后直接访问 `http://host:port/`

### 私域知识库 RAG（核心亮点）

支持三种数据源：

| 来源 | 接入方式 | 适用场景 |
|------|---------|---------|
| 飞书知识库 | MCP 协议 + 增量同步 | 产品文档、流程规范 |
| 飞书云空间 | MCP 协议 + 文件夹递归 | 项目文档、临时材料 |
| 本地文档 | 目录扫描（md/txt/sql/yaml） | SQL 配置、不便上传的内容 |

检索策略：

```
用户提问
   │
   ├─▶ 向量检索（ChromaDB + text2vec-base-chinese）  ─┐
   │      语义匹配，找"意思相近"的文档                │
   │                                                  ├─▶ RRF 融合排序 ──▶ 注入 Prompt
   └─▶ BM25 检索（jieba 分词）                       ─┘
          关键词精确匹配，找"字面相同"的文档
```

关键设计：
- **增量更新**：基于飞书文档 `edit_time` 比对，只拉变更的
- **保留目录结构**：本地缓存按飞书原始路径组织，方便排查和手动删除
- **附带原文链接**：回答时给出飞书原文 URL，可一键跳转
- **大文档分块**：> 2000 字符自动分块索引，提升检索质量

### 多 AI 模型支持

| Provider | 适用场景 |
|----------|---------|
| Anthropic Claude | 长上下文、文档问答效果最优 |
| OpenAI 兼容 | GPT / Ollama / LM Studio / Azure |
| Dify | 已有 Dify 应用，直接对接 |
| 本地模型 | 数据敏感场景，Ollama 私有化部署 |

### 生产级部署

| 维度 | 实现 |
|------|-----|
| 容器化 | Docker + docker-compose，CPU/GPU 双镜像 |
| GPU 加速 | ONNX Runtime CUDA，Embedding 速度 5-10×，支持多卡轮询 |
| 高并发 | 事件队列 + 线程池，多用户同时提问 |
| 可观测 | Trace ID 全链路 + 心跳日志 + 性能埋点 |
| 健壮性 | 信号处理、全局异常捕获、graceful shutdown |
| 运维 | `service.sh start/stop/restart/status/debug/monitor` |

## 七、性能数据

### 检索性能

| 场景 | CPU 模式 | GPU 模式（单卡 P40）| 加速比 |
|------|---------|---------------------|--------|
| 索引 1000 篇文档 | ~120s | ~15s | 8× |
| 单次查询 | ~500ms | ~100ms | 5× |

### 端到端响应

| 阶段 | 耗时 | 说明 |
|------|-----|------|
| Webhook 接收 → 入队 | < 50ms | Flask + Waitress |
| 文档检索（命中缓存） | 100-300ms | 向量 + BM25 并行 |
| LLM 调用（Claude Sonnet） | 2-5s | 取决于上下文长度 |
| 飞书回复 API | < 500ms | 官方 API |
| **总计** | **3-6s** | 用户感知良好 |

### 并发能力

- 默认 5 个 worker 线程，可配置
- 队列 100 容量，自动削峰
- 实测同时 10 用户提问，平均响应 4.2s（无队列堆积）

## 九、安全设计

| 维度 | 设计 |
|------|------|
| 配置安全 | API Key 不入 Git；config.yaml 强制 .gitignore |
| 数据隐私 | 消息仅本地处理；日志不记录完整内容（仅长度+摘要）|
| 非侵入式 | 仅用官方 API；不修改 IM 客户端、不 hook 系统 API |
| 权限可控 | 群组/用户白名单；用户可随时停用 |
| 网络隔离 | 支持禁用系统代理；支持私有化大模型（Ollama 等） |

## 十、如何接入

```bash
# 1. 拉代码
git clone <repo-url> && cd AI-Personal-Assistant

# 2. 配置（飞书 App ID + 大模型 API Key）
cp config.example.yaml config.yaml
vim config.yaml

# 3. Docker 部署（推荐）
cd docker && docker-compose up -d ai-assistant-gpu

# 4. 飞书开放平台填 Webhook URL
# https://your-domain.com/webhook/feishu
```
